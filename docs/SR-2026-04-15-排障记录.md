# SR（2026-04-15）排障记录

## 1. 结论

本次 `SR` 数据缺失不是单一原因，而是三层问题叠加：

1. 当前生产使用的 Chainstack HTTP 节点，不具备本项目这套“历史时间窗回扫 + 自动识别内盘地址”所依赖的历史日志与 trace 能力。
2. 新 `SR` 的链上买入结构，和此前成功入库的老项目结构不同。
3. 当前生产解析器原先默认“必须存在 `fee_addr` 那条腿”，因此会把 `SR` 这类 `tax-only` 交易全部过滤掉。

## 2. 已证明的事实

### 2.1 新旧 SR 不是同一个链上项目

来自 `SignalHub-main/signalhub.db` 的直接证据：

旧 `SR`：

- `project_id = 51547`
- `name = Strike Robot`
- `token_address = 0xe5B99C2E1672D189C43A5B68AB7699270C3339C0`
- `internal_market_address = 0xaf76783354ff1c6021629d916dac9fd31a7342fd`
- `launch_time = 2026-04-06T19:00:00.000Z`
- `subscription_status = confirmed`

新 `SR`：

- `project_id = 70972`
- `name = Strike Robot`
- `token_address = 0x10c56F005a379f8eAfc88ff5c3f40d30F0031AC9`
- `internal_market_address = 空`
- `launch_time = 2026-04-13T03:00:13.000Z`
- `subscription_status = error`
- `last_error = 403 Forbidden`

因此：

- 旧 `SR` 与新 `SR` 的 `project_id` 不同
- 旧 `SR` 与新 `SR` 的 `token_address` 不同
- 两者不是“同一项目延期”，而是“同名但不同链上对象”

### 2.2 手动填入的新 SR 内盘地址是正确的

生产主库 `managed_projects` 当前记录：

- `signalhub_project_id = 70972`
- `token_addr = 0x10c56f005a379f8eafc88ff5c3f40d30f0031ac9`
- `internal_pool_addr = 0x745c62e112435afcbd942504002d1ddd305ce0db`
- `status = ended`
- `is_watched = 1`
- `collect_enabled = 1`
- `backfill_enabled = 1`

对新 `SR` 真实交易 receipt 的核对结果：

- 买家会把支付代币打到 `0x745c62...`
- 同时 `0x745c62...` 会把 `SR` token 打给买家

因此：

- `0x745c62E112435AfcbD942504002d1ddd305Ce0dB` 是成立的内盘候选地址
- “地址填错”不是本次缺数根因

### 2.3 当前 RPC plan 无法满足回扫与自动识别需求

当前线上使用/测试过的 3 条 Chainstack HTTP 节点：

- 完整 endpoint 属于运行密钥，不在文档中记录；请从服务器 `config.json` 或环境变量读取。

对三条节点的统一实测结果：

- `eth_blockNumber`：`200`
- `eth_getBlockByNumber(历史块)`：`403`
- `eth_getLogs`：`403`
- `debug_traceTransaction`：`403`

返回的错误文本为：

- `Archive, Debug and Trace requests are not available on your current plan`

另外，生产 `scan_jobs` 里还真实出现过：

- `-32005`
- `You've reached your monthly quota of Request Units (RUs)`

因此：

- 当前问题既包含“节点功能不够”，也包含“某条回扫节点额度打满”
- 不是节点完全失效，而是当前 plan 无法满足本项目所需方法集

### 2.4 新 SR 的链上买入结构与旧项目不同

对比老项目 `F007` 的成功样本：

- 买家 -> `internal_pool_addr`
- 买家 -> `fee_addr`
- `internal_pool_addr` -> 买家（项目 token）

对比新 `SR` 的真实样本：

- 买家 -> `internal_pool_addr`
- 买家 -> `tax_addr`
- `internal_pool_addr` -> 买家（项目 token）

关键差异：

- `F007` 存在 `buyer -> fee_addr`
- `SR` 不存在 `buyer -> fee_addr`
- `SR` 是 `tax + pool`，不是原有解析器假设的 `fee + pool`

### 2.5 原解析器会漏掉 SR 这类交易

旧解析器核心约束：

- 若 `fee_v = 0`，则 `spent_v_est = fee_v / fee_rate = 0`
- `spent_v_est <= 0` 时，整笔交易直接丢弃

对 `SR` 样本实测：

- 在旧规则下，同一笔 `SR` 交易 `parsed_events = 0`
- 在增加 fallback 后：
  - `parsed_events = 1`
  - `fee_v = 0`
  - `tax_v > 0`
  - `spent_v_est = spent_v_actual`
  - `anomaly = false`

因此：

- `SR` 数据为 `0` 的重要原因之一，是原解析器对 `tax-only` 结构不兼容

## 5. 当前最准确的因果链

1. 新 `SR` 不是旧 `SR` 的延期版，而是新的链上对象。
2. `SignalHub` 自动识别新项目内盘地址时，被当前 RPC plan 的 `403` 卡住。
3. 管理员手动补入了正确的内盘地址。
4. 主项目自动回扫又遇到两层 RPC 问题：
   - 一阶段：`RU quota exceeded`
   - 二阶段：`Archive, Debug and Trace requests are not available on your current plan`
5. 即使绕过时间窗回扫，直接使用真实 tx hash 导入，旧解析器也会把 `SR` 的 `tax-only` 交易过滤掉。
6. 因此最终表现为：
   - 项目已开始
   - 内盘地址正确
   - 但主项目仍为 `0 events`

## 6. 当前已做的技术处理

- 本地与生产均已加入回扫 RPC fallback：
  - 若独立回扫节点 RU 打满，则回退到主 HTTP 节点
  - 若 `eth_getLogs` 不可用，则尝试走逐块扫描 fallback
- 已实现一版解析器兼容：
  - 若 `fee_v = 0` 且 `spent_v_actual > 0`
  - 则允许以 `spent_v_actual` 作为 `spent_v_est` fallback
- 已使用公开 Base RPC 完成 `SR` 的一次性历史重放：
  - `https://base-rpc.publicnode.com`
  - 用于补充历史 `block timestamp / receipt` 查询，绕开当前 Chainstack plan 的历史块限制

注意：

- 上述兼容逻辑已在生产环境抽样验证成功，并已完成 `SR` 的 `447` 笔历史重放
- 当前未解决的是“生产自动回扫长期仍依赖可用的历史日志 / trace RPC”，不是 `SR` 这次的单次补数
