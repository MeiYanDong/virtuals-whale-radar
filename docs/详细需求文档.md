
## 一、项目概述

### 1.1 项目名称
Virtuals Whale Radar

### 1.2 项目定位
基于历史项目 Virtuals-Launch-Hunter（VLH，当前已整合为 Virtuals Whale Radar 主服务）与 SignalHub（SHV）进行整合，构建一个面向管理员与普通用户的 Virtuals 发射项目追踪平台。

该平台的核心目标是：

通过 SHV 提供的项目信息与时间数据，自动驱动 VLH 的追踪流程，减少人工配置成本，并将追踪结果以可视化方式提供给管理员和普通用户查看。

### 1.3 项目目标
实现“项目发现 → 自动配置 → 自动追踪 → 数据展示 → 用户自定义钱包观察”这一完整链路，形成统一的平台化产品。

---

## 二、现有项目基础

### 2.1 Virtuals-Launch-Hunter（VLH，历史名）
VLH 当前能力包括：

用于追踪 Virtuals 待发射项目  
用户填写追踪代币地址后，后端执行打新阶段的数据采集、计算、追踪与分析  
前端展示以下核心数据：

分钟消耗 SpentV（柱状图）  
大户榜  
我的钱包持仓

### 2.2 SignalHub（SHV）
SHV 当前能力包括：

持续爬取 Virtuals 官方发布的待发射项目信息  
追踪相关项目，并在项目发射的第一时间返回关键地址信息  
通过内置接口向外提供 API，供其他程序调用

SHV 返回的数据至少应包括：

项目基础信息  
发射时间相关信息  
主交易池地址  
内盘地址  
项目对应代币地址（若有）

---

## 三、总体建设目标

整合 VLH 与 SHV，形成双端平台：

管理员端  
用于项目接入、自动参数填充、追踪任务管理、回扫与实时更新控制

用户端  
用于查看即将到来的项目列表、浏览项目打新过程数据、管理个人追踪钱包

---

## 四、角色定义

### 4.1 管理员
管理员拥有平台配置与任务管理权限，包括但不限于：

查看 SHV 同步的即将到来项目  
配置或修正项目参数  
触发或管理自动采集、回扫、实时更新任务  
查看项目追踪结果  
管理系统级追踪逻辑

### 4.2 普通用户
普通用户仅拥有浏览与个人钱包管理权限，包括：

注册与登录  
浏览项目列表  
查看项目追踪数据  
添加、编辑、删除个人追踪钱包  
为钱包设置自定义名称

普通用户不具备任何系统配置、任务控制或项目编辑权限。

---

## 五、功能需求

## 5.1 管理员端

### 5.1.1 页面定位调整
将现有 VLH 前端页面改造为管理员页面，保留现有核心数据展示能力，同时增加项目管理能力。

### 5.1.2 即将到来的项目列表
管理员页面新增“即将到来的项目列表”模块。

数据来源：
通过 SHV 提供的 API 获取

展示内容建议包括：
项目名称  
项目状态  
预计开始时间  
预计结束时间  
主交易池地址  
内盘地址  
代币地址  
是否已完成自动配置  
当前追踪状态

### 5.1.3 自动参数填充
当 SHV 返回有效数据时，系统应自动填充 VLH 所需核心参数，包括：

scanStartUtc8  
scanEndUtc8  
追踪代币地址  
内盘地址

若 SHV 未返回完整数据，或自动填充失败，则系统应：

在管理员页面明确提示  
允许管理员手动补充参数  
保存管理员补充后的配置结果

### 5.1.4 自动执行任务
当项目参数完整后，系统应按预定流程自动执行以下任务：

采集  
回扫  
实时更新

系统需要支持根据 SHV 返回的时间信息自动触发任务，无需管理员重复手动操作。

### 5.1.5 管理员端数据查看
管理员端继续保留并展示以下 VLH 核心数据：

分钟消耗 SpentV（柱状图）  
大户榜  
我的钱包持仓

其中“我的钱包持仓”在管理员端延续现有管理逻辑，用于管理员自身观察。

---

## 5.2 用户端

### 5.2.1 注册与登录
用户端必须具备完整的注册与登录流程。

未登录用户访问用户页面时，应自动重定向至注册页或登录页。

### 5.2.2 项目列表页
用户端需要提供“即将到来的项目列表”页面。

列表数据需与管理员端保持一致。  
实现方式可选以下两种之一：

方案 A：直接调用管理员端所使用的数据源  
方案 B：直接调用 SHV API 获取项目列表

最终实现时建议保证“单一数据源”原则，避免管理员端与用户端列表不一致。

### 5.2.3 项目详情页
用户点击项目列表中的某个项目后，应跳转至该项目的打新过程数据页面。

页面展示内容包括：

分钟消耗 SpentV（柱状图）  
大户榜  
我的钱包持仓

其中“我的钱包持仓”在用户端应基于当前登录用户已配置的钱包进行展示。

### 5.2.4 用户侧边栏与设置页
用户页面采用双栏布局：

中间主栏：
展示即将到来的项目列表，或进入具体项目详情

左侧栏：
展示用户基本信息  
展示设置按钮

点击设置按钮后，进入用户设置页面。

### 5.2.5 用户追踪钱包管理
用户设置页支持以下功能：

添加追踪钱包  
删除追踪钱包  
编辑追踪钱包名称  
为钱包地址设置自定义名称

钱包追踪逻辑与管理员页面保持一致，即：

系统基于配置的钱包地址，在项目详情中展示对应的钱包持仓或相关数据。

## 七、数据流与系统逻辑

### 7.1 数据流主链路
1. SHV 持续抓取 Virtuals 待发射项目信息  
2. SHV 通过 API 输出项目基础数据与关键地址信息  
3. 管理员端读取 SHV API，同步“即将到来的项目列表”  
4. 系统自动提取并填充 VLH 所需参数  
5. 当参数完整时，自动执行采集、回扫、实时更新  
6. VLH 输出可视化分析数据  
7. 管理员端与用户端分别按权限展示数据

### 7.2 异常兜底逻辑
若 SHV 返回数据不完整，则：

项目进入“待人工补充”状态  
管理员收到缺失提示  
管理员手动补充后，系统恢复自动流程

## 九、非功能需求

### 9.1 一致性要求
管理员端与用户端的项目列表必须保持一致。  
同一项目的基础信息、状态信息、时间信息应统一来源，避免双端不一致。

### 9.2 可扩展性要求
系统设计需支持后续扩展，包括但不限于：

更多项目来源接入  
更多链上指标展示  
更多用户自定义观察维度  
消息提醒能力接入

### 9.3 易维护性要求
管理员端与用户端应尽量复用相同的数据模型、接口结构与追踪逻辑，降低维护成本。

## 十、建议的开发优先级

### 第一阶段
完成 SHV 与 VLH 的数据打通  
完成管理员端项目列表与自动参数填充  
完成自动采集、回扫、实时更新流程

### 第二阶段
完成用户注册、登录与权限控制  
完成用户端项目列表与项目详情页

### 第三阶段
完成用户钱包管理  
优化双端 UI 与状态提示  
完善异常处理与日志体系

## 十一、待确认事项

以下问题建议在正式开发前进一步明确：

1. 用户端项目列表最终采用哪一种数据源  
是直接读取 SHV API，还是通过平台后端统一转发

2. 用户注册登录采用何种认证方式  
例如邮箱密码、验证码、JWT、Session 等

3. “我的钱包持仓”在用户端的计算口径  
是否仅统计当前登录用户配置的钱包  
是否支持多钱包聚合

4. 自动任务调度机制
采用服务端定时任务、消息队列，还是事件驱动方式

5. 管理员是否需要手动覆盖 SHV 自动同步结果
例如修改 scanStartUtc8、scanEndUtc8 或替换地址信息
